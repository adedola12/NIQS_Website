3 Hosting & Performance

3.1 API server (Render) — upgrade off the free plan

This is the main cause of slowness today
On Render's free plan, the server goes to sleep after ~15 minutes of inactivity. The next visitor then waits 30–60 seconds while it wakes up — pages, images and admin actions all stall on that first request. The free plan also has limited memory/CPU and no persistent disk.

Recommended: upgrade the Render service to a paid always-on plan (Starter tier). This removes cold-starts entirely — the single biggest, cheapest speed win. Also set NODE_ENV=production and choose a region close to Nigeria/your database.

3.2 Web client (Vercel)

Already served from a fast global CDN — no speed concern. Actions: attach the official domain (§4) and confirm the Vercel plan is appropriate for an organisation (the free Hobby tier is for non-commercial use; confirm whether Pro is required).

3.3 Database (MongoDB Atlas)

  • Confirm the cluster tier. The free M0 tier has limited performance and no automated backups — a small paid tier is recommended for a production institute site.
  • Lock Network Access to Render's egress addresses (or secure it properly) and keep a strong database user/password.
  • Pick a region close to the API server to minimise latency.

3.4 Media storage — confirm cloud keys in production

Why this matters
Uploaded images/files (news photos, flyers, journals, brand assets) stream to Cloudinary (<10 MB) or Cloudflare R2 (≥10 MB). If those credentials are not set in production, the system falls back to the server's local disk — which Render wipes on every redeploy, so uploaded media would silently disappear. Action: verify the Cloudinary/R2 keys are present in the Render production environment.

4 Domain, DNS & SSL

The site currently lives on niqs-website.vercel.app. For launch it needs the official NIQS address.

  • Decide the official domain — e.g. niqs.org.ng vs niqsng.org. (This also affects email, social-preview links and the portal integration, so confirm one canonical domain.)
  • Provide access to the domain registrar / DNS so records can be pointed to Vercel.
  • Point DNS (A / CNAME) to Vercel — SSL (https) is provisioned automatically.
  • Add the domain to the server's allowed origins (CLIENT_URL) so the site can talk to the API.
  • Update the social-preview tags (og:url, og:image) — currently they point to the vercel.app address.
  • Align the email sending domain (SPF / DKIM) with §6.

7 Payments Decision needed

The Payments and Membership pages currently display fee tables with a placeholder "payment coming soon" button. To go live with online payment, NIQS must choose one of:

OptionWhat it needs
A. Payment gateway on the website (Paystack / Flutterwave — Nigeria)A merchant account + API keys. Best if the website collects dues/exam/event fees directly.
B. Route payments through the membership portalThe portal owns dues; the website links to its checkout (see the companion portal document, §3.9). Best if the portal already handles money.

Recommendation: if the membership portal will own dues/renewals, use Option B for member payments and reserve Option A only if the website needs to take standalone payments (e.g. public event tickets).

8 Membership Portal Integration

The member experience — login, the registered-QS directories, dues status, CPD, ID card — depends on the membership portal's API, which the portal team is building. This is fully specified in the companion document "Membership Portal — API Integration Requirements".

  • The website's portal client is already built and env-gated — it stays safely inert until the portal is ready, so it does not block a content launch.
  • When the portal team delivers, we simply set the PORTAL_* credentials and the member features come alive.
  • A content/marketing launch can go ahead now; the full member portal switches on once the API is available.

9 Security & Accounts (Pre-Launch Hardening)

  • Remove the test/seed endpoints. A setup endpoint currently lets anyone create a test member account (with credentials visible in the code). This must be removed and the test account deleted before launch. The one-time admin-seed endpoint should also be removed once real admins exist.
  • Set strong, unique secrets in production (JWT_SECRET, JWT_REFRESH_SECRET) — not the sample values.
  • Set NODE_ENV=production so login cookies are issued securely (HTTPS-only).
  • Create real admin accounts for Secretariat staff with the correct roles (national, state-chapter, WAQSN/YQSF sub-admins) and hand over credentials securely. Remove any test admins.
  • Verify sign-in works on the production domain (the website and API run on different addresses — confirm member/admin login and sessions behave correctly once the official domain is live).
  • Decide on the "Built by ADLM" credit badge shown on every page — keep as a credit, or remove for launch.
  • Optional hardening: rate-limiting on login and public forms; basic abuse protection on file uploads.

10 SEO, Analytics & Legal

  • Add web analytics (Google Analytics 4 or Plausible) — none is installed yet, so visits aren't measured.
  • Add robots.txt and a sitemap.xml; submit the site to Google Search Console for indexing.
  • Update social-preview tags (og:url, og:image) to the official domain; confirm the favicon.
  • Publish a Privacy Policy and Terms of Use — important given member data, contact/flyer forms and (future) payments; align with Nigeria's NDPR data-protection rules.
  • Add a brief cookie notice if analytics is enabled.

11 Backups & Monitoring

  • Enable automated database backups (a paid Atlas tier includes them) so member and content data is recoverable.
  • Set up uptime monitoring with alerts (e.g. UptimeRobot / Better Stack) so any outage is noticed immediately.
  • Optional: error tracking (e.g. Sentry) to catch issues proactively.

12 Pre-Launch QA Checklist

  • Test every form end-to-end: contact, event registration, flyer request, member sign-up / login / password reset.
  • Test admin create/edit/delete for each content type.
  • Check the site on mobile and across major browsers.
  • Check for broken links and that all images load from cloud storage (not the temporary local path).
  • Confirm email delivery once SMTP is set (§6), and the payment flow once the gateway is chosen (§7).
  • Run a performance/Lighthouse pass after the hosting upgrade.

13 Onboarding & Handover to NIQS

  • Admin training session — walk the Secretariat through managing content (news, events, leadership, chapters, journals, etc.) via the admin panel.
  • Secure credential handover — admin logins, plus access to hosting (Vercel, Render), the database and media accounts as agreed.
  • Provide a short "how to update each section" guide alongside this document and the portal API document.
  • Agree ownership of ongoing content updates and a support / maintenance arrangement (point of contact, response times, update cadence).

14 Go-Live Runbook & Rollback

Recommended order of cut-over:

StepAction
1Approve hosting budget; upgrade Render (always-on) and the Atlas tier.
2Set all production environment variables (database, secrets, NODE_ENV, CORS/domain, media, email).
3Configure the official domain + DNS + SSL; update CORS and social tags.
4Populate real content and images; confirm demo/sample data is replaced.
5Harden security: remove seed endpoints, set secrets, create real admin accounts.
6Add analytics, sitemap, and legal pages.
7Full QA pass (§12).
8Announce launch and monitor closely for the first days.
Rollback safety
Both Vercel and Render keep previous deployments — if anything goes wrong, we can roll back to the last good version almost instantly. Keep DNS TTL low during cut-over for a fast switch.
