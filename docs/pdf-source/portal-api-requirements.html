1 Background & Ownership Boundary

Two systems make up the NIQS digital platform. The public website (built by ADLM Studio) handles public content, events, flyers and the member-facing pages. The Membership Portal (portal.niqsng.org, built by your team) is the system of record for member identity, the registered-QS register, dues and member records. For NIQS members to experience one seamless platform, the website must read from — and write a little back to — the portal through a shared API.

1.1 Who owns what

DomainOwnerNotes
Member identity & login credentialsPortalWebsite uses SSO; never stores member passwords.
Membership record (number, type, status, expiry)PortalWebsite reads via API.
Registered-QS directory (individuals)PortalWebsite renders search & segmented views.
Dues, payments, invoicesPortalWebsite links to checkout / reads standing.
CPD ledger, exam results, ID cardPortalWebsite displays; posts CPD earned at its events.
Public content (news, events, chapters, exco, partners, brand)Website—

QS Firm (company) directoryWebsiteToday website-managed; could move to portal — see D9.

2 Global API Conventions

These apply to every endpoint in Section 3 so our client needs only configuration, not per-endpoint rewrites.

ItemConvention
Base URLhttps://portal.niqsng.org/api/v1  (confirm — see D7). One base per environment: staging + production.
AuthAuthorization: Bearer <PORTAL_API_KEY> — one key per environment, server-to-server only, never sent to the browser.
TransportHTTPS only. JSON request & response, UTF-8.
PaginationList endpoints accept page (1-based) & pageSize; return total, page, pageSize.
IdempotencyWrite-backs (CPD, library) are idempotent on a reference field — retries must not double-count.

3 Required Endpoints — the Portal builds these

3.1 Member login & Single Sign-On — link to the portal you built

A member clicks Member Login on the website and authenticates against your portal (single source of truth for credentials). We then mint a short-lived website session. Recommended: redirect + code exchange.

REDIRECTGET /sso/authorizeP2
Browser redirect to your hosted login. On success you redirect back to our callback with a one-time code.
Query
?client_id=NIQS_WEBSITE &redirect_uri=https://niqs.org.ng/login/portal/callback
&state=<csrf> &scope=profile
POST/sso/tokenP2
Our server exchanges the one-time code (Bearer API key) for the member + a session handle.
Body → Response
{ "code": "<one-time>" }   →   { "member": { …§3.2 }, "sessionId": "…", "expiresIn": 3600 }
If your portal already speaks OIDC, we can consume standard Authorization Code + PKCE instead — tell us which (D1).
POST/auth/introspectP2
Validate any portal-issued token the browser presents.
Body → Response
{ "token": "…" }   →   { "active": true, "member": { …§3.2 } }

3.4 Registered-QS directory search — powers four views from one endpoint

This single endpoint delivers four of NIQS's requested lists
All registered QS, QS in each state chapter, YQSF members and WAQSN (female) members are the same search with different filters. The portal owns the individual register; the website renders the views. (The website's existing /search-qs-firms covers companies only — the individual register is new and yours.)
GET/members/searchP1
Search / list the register with filters. Return only publicly displayable fields (no email/phone in lists).
Query parameters
?q=             // name or membership number (free text)
&state=Lagos      // → "QS in this state chapter"
&chapter=lagos    // chapter slug or id (see D3)
&type=corporate   // membershipType
&status=active    // default active
&gender=female    // → WAQSN view
&forum=yqsf|waqsn // portal applies the segment rule server-side (§7)
&sort=name|recent &page=1 &pageSize=24
200 Response
{
  "results": [
    { "membershipNumber": "NIQS/12345", "fullName": "QS Jane Doe",
      "membershipType": "corporate", "status": "active",
      "state": "Lagos", "chapter": "Lagos", "gender": "female",
      "isYqsf": false, "firmName": "Greenfield QS Ltd", "photoUrl": null }
  ],
  "total": 134, "page": 1, "pageSize": 24
}
NIQS list requestedCall
All registered QS (national register)/members/search (no segment filter)
Registered QS in each state chapter/members/search?chapter=<slug>  or  ?state=<state>
YQSF members (Young QS Forum)/members/search?forum=yqsf
Female members for WAQSN/members/search?forum=waqsn  or  ?gender=female

3.9 Payments & member purchases — dues · exams · events · store

The website's /payment and /membership pages are static placeholders today. The portal owns money.

5 Webhooks the Portal calls on the Website

So the website can invalidate sessions/caches and reflect changes in real time. Sign each request with a shared secret: X-NIQS-Signature: sha256=<hmac(secret, rawBody)>.

Website endpoint (we build)Event payload
POST /api/portal/webhooks/membership { event: "member.updated|member.suspended|member.renewed", membershipNumber, status, occurredAt }
POST /api/portal/webhooks/payment { event: "payment.succeeded", membershipNumber, item, amount, ref, occurredAt }
For SSO return, the website also exposes GET /login/portal/callback (browser) and POST /api/auth/portal/exchange (server). We will share exact URLs & the webhook secret during integration.

7 Segment Rules — YQSF & WAQSN

These business rules must be defined by NIQS and computed by the portal (so the forum= filter, the isYqsf flag and the counts are consistent everywhere).

YQSF — Young Quantity Surveyors Forum
The website states membership is for QS under 40 years of age or within their first ten years of professional practice, and members auto-exit at 40. To compute forum=yqsf / isYqsf / the yqsf count, the portal must store date of birth and/or year of first registration/practice. Confirm the exact rule (D4).
WAQSN — Women Association of QS Nigeria
Is WAQSN all female members (derive from gender), or a separately enrolled subgroup some women opt into? If the latter, expose it as forum=waqsn; otherwise gender=female suffices. Either way the portal must store gender. Confirm (D4).

8 Security & Operational Requirements

  • API keys live only in the website server environment — never shipped to the browser. All portal calls are server-side proxied.
  • HTTPS everywhere; optional IP allowlist for server-to-server traffic.
  • Webhooks signed with a shared secret (HMAC-SHA256 over the raw body).
  • Token TTLs: SSO session ≤ 1 hour (refresh via re-exchange); one-time codes ≤ 5 minutes.
  • PII minimisation: directory search returns only public fields; email/phone require an authorised profile call.
  • CORS: allowlist the website origins (production + Vercel preview deployments) for any browser-facing endpoints.
  • Rate limiting + idempotency: agree 429 behaviour + retry/backoff; CPD & library posts idempotent on reference.
  • Environments: provide staging and production base URLs and a separate API key for each.

9 Open Decisions & Phased Delivery

9.1 Decisions needed from NIQS / the Portal team

D1SSO style — redirect + code exchange (recommended) or full OIDC? And is the member area website-rendered or a deep link into your portal UI?
D2Membership-number format — confirm canonical form (e.g. NIQS/12345).
D3Chapter matching key — match members to chapters by chapter id/slug or by state string?

D6Job applications — stored website-side (recommended) or posted to the portal? Will you expose member CVs?

D8ID card — does the portal render the card (return a URL) or return fields for the website to render?
D9QS Firm directory — stays website-managed or moves to the portal?
